# Техническое задание — EnergyAudit Bot

**Версия:** 1.0  
**Назначение:** Производственная практика по направлению .NET-разработки  
**Период реализации:** 4–5 недель

---

## 1. Общая информация

**Название:** EnergyAudit Bot  
**Тип:** Telegram-бот с ИИ-анализом  
**Цель:** Помочь владельцам жилья и офисов снизить расходы на электроэнергию через ежедневный мониторинг потребления и персональные рекомендации.

**Ключевое отличие от аналогов:**  
Не разовый аудит — а постоянный мониторинг через быстрый ежедневный чек-ин (1 минута, только кнопки). ИИ анализирует реальную историю пользователя, а не абстрактные советы.

---

## 2. Целевая аудитория

- Владельцы квартир и домов, желающие снизить счета за свет
- Арендаторы офисов, ответственные за коммунальные расходы
- Пользователи, интересующиеся экологией и осознанным потреблением

---

## 3. Функциональные требования

### 3.1 Онбординг (первый запуск)

При первом `/start` бот проводит пользователя через настройку профиля:

1. Тип жилья — Квартира / Дом / Офис (кнопки)
2. Площадь — в кв. м (текстовый ввод)
3. Количество жильцов (кнопки: 1 / 2 / 3 / 4+)
4. Тариф за электричество — ₸/кВт·ч (текстовый ввод или выбор региона)
5. Основные приборы — выбор из списка с мощностью (мультивыбор)

После онбординга пользователь сразу может сделать первый чек-ин.

---

### 3.2 Ежедневный чек-ин `/audit`

Главная функция бота. Сценарий:

```
1. Бот показывает список приборов пользователя (Inline Keyboard)
   "Что работало сегодня? Отметь нужное:"
   [✅ Кондиционер] [Стиральная машина] [Духовка] [Готово]

2. Для каждого отмеченного прибора — выбор времени работы:
   "Кондиционер: сколько часов?"
   [0.5ч] [1ч] [2ч] [4ч] [8ч+]

3. После всех приборов бот показывает итог дня:
   ⚡ Сегодня: 7.4 кВт·ч = 167 ₸
   
   Кондиционер     — 65% (4.8 кВт·ч)
   Стиральная маш. — 27% (2.0 кВт·ч)
   Освещение       — 8%  (0.6 кВт·ч)
   
   📉 Вчера было 9.1 кВт·ч — на 19% меньше, молодец!
```

**Ограничения:**
- Один чек-ин в день на пользователя
- Если пользователь не сделал чек-ин — вечером приходит напоминание

---

### 3.3 Статистика `/stats`

Пользователь выбирает период: За неделю / За месяц

Бот показывает:

```
📊 Статистика за неделю (26 мая — 1 июня)

Потреблено:   49.3 кВт·ч
Расходы:      1 118 ₸
В среднем:    7.0 кВт·ч/день

Прошлая неделя: 61.8 кВт·ч
Экономия:       ▼ 20% (−283 ₸) 🎉

Главный потребитель: Кондиционер (58%)
```

---

### 3.4 Персональный совет от ИИ `/tips`

Один раз в день пользователь может запросить совет.

ИИ получает:
- Профиль жилья
- Историю чек-инов за 7 дней
- Прибор с наибольшим потреблением

ИИ возвращает конкретный совет с цифрами:

```
💡 Совет дня

За неделю кондиционер потребил 58% всей энергии.
Попробуй установить 26°C вместо 22°C — это снизит
потребление на ~30% без потери комфорта.

Потенциальная экономия: ~850 ₸/месяц
```

---

### 3.5 История потребления `/history`

Последние 7 / 30 дней в виде списка:

```
📅 История потребления

01 июня — 7.4 кВт·ч (167 ₸)
31 мая  — 9.1 кВт·ч (206 ₸)
30 мая  — 6.8 кВт·ч (154 ₸)
...
```

---

### 3.6 Профиль `/profile`

Просмотр и редактирование данных жилья и приборов:

```
🏠 Мой профиль

Тип:      Квартира
Площадь:  54 кв.м
Тариф:    22.68 ₸/кВт·ч
Жильцов:  2

Мои приборы:
  • Кондиционер (1.5 кВт)
  • Стиральная машина (2.0 кВт)
  • Холодильник (0.15 кВт)

[✏️ Изменить] [➕ Добавить прибор]
```

---

### 3.7 Вечерние напоминания

Каждый день в 20:00 (по часовому поясу региона пользователя) бот отправляет напоминание тем, кто не сделал чек-ин:

```
👋 Не забудь отметить потребление за сегодня!

[⚡ Сделать чек-ин]
```

## 4. Нефункциональные требования

| Требование | Значение |
|---|---|
| Время ответа бота | ≤ 3 сек (без ИИ) |
| Время ИИ-ответа | ≤ 15 сек |
| Одновременные пользователи | 100+ |
| Хранение данных | Бессрочно (чек-ины и профили) |
| Хранение токенов/ключей | Только через переменные окружения |
| Доступность | ≥ 99% (Webhook + Docker restart policy) |

---

## 5. Технологический стек

| Компонент | Технология | Обоснование |
|---|---|---|
| Backend | ASP.NET Core 8, Minimal API | Современный, быстрый, подходит для практики |
| Telegram | Telegram.Bot + Webhook | Надёжнее Polling в production |
| БД | EF Core + PostgreSQL | Гибкость + JSON-поля для usages |
| ИИ | Semantic Kernel + OpenAI | Абстракция над LLM-провайдерами |
| Логирование | Serilog | Структурированные логи |
| Валидация | FluentValidation | Читаемые правила валидации |
| Тесты | xUnit + Moq | Стандарт в .NET-экосистеме |
| Деплой | Docker + Docker Compose | Воспроизводимость окружения |
| API Docs | Scalar | Современная альтернатива Swagger UI |

## 9. Риски

| Риск | Вероятность | Мера снижения |
|---|---|---|
| Лимиты бесплатных LLM API | Высокая | Кэшировать результат на 24ч; поддержка нескольких провайдеров |
| Сложности с Webhook (нет публичного домена) | Средняя | Использовать ngrok при разработке; polling как fallback |
| Неточность расчётов потребления | Средняя | Мощности приборов — среднестатистические; предупреждать пользователя |
| Потеря состояния диалога при рестарте | Низкая | Хранить незавершённый диалог в Redis (опционально) |
